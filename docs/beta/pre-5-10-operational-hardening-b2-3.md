first 1–2 testers to **5–10 testers**, and records a **go/no-go**. It is operational, not

> retained historical residue with no beta-window recurrence.**

  and `movie_title` (2, last ~2026-05-20), attached to **2 heavy dev/QA persons** (446 and 891

- **Why it is historical / not a beta issue:** both events pre-date the 2026-06-11 tester window

  `result_kind`, `query_length_bucket` (coarse), `movie_id` — no raw query, no `movie_title`;
  `analytics.js` `identify` is user-id-only; replay masks all inputs + all text; the fail-closed
  `analyticsPrivacy.test.js` source-scan blocks `query`/freeform keys in any event.
- **Why not deleted now:** the available MCP tool (`persons-bulk-delete`) is **person-level only**
  — deleting these persons' events would remove **~1,337 dev events**, far exceeding the authorized
  2-event scope, and is irreversible. Surgical, property-filtered event deletion is **dashboard-only**.
- **Manual operator step (surgical):** PostHog → **Data management → Delete events**, filtered to
  `event = search_performed` where `query` is set OR `movie_title` is set; or use PostHog's async
  data-deletion API with the same filter. (A heavier option — `persons-bulk-delete` with
  `delete_events:true` on the 2 dev persons — exists but over-deletes dev analytics; use only if
  losing that dev history is acceptable.)
- **When:** before public beta/production (required); before 5–10 is **recommended** (low blast
  radius: dev personas, not testers). No app-side PostHog person-delete exists today.

## 8. New findings this phase

### 8a. PostHog client analytics is dark in production — **P2 (observability; fails safe)**
PostHog received **8 events in the last 7d, all dated 2026-06-04, and 0 on the tester days
(06-05 → 06-11)**, while `recommendation_impressions` shows **997 rows / 5 distinct users** in the same
window. The entire wired beta funnel (`home_opened`, `discover_opened`, `recommendation_shown/_saved`,
`people_*`, `onboarding_started`) has **never appeared in the PostHog taxonomy**. CSP (allows
`*.posthog.com`) and analytics opt-out (0 of 2 settings opted out) are ruled out. **Most likely cause:**
`VITE_POSTHOG_KEY` is unset/empty in the Cloudflare Pages **production** build (`initAnalytics()` silently
no-ops without the key) — **not verifiable from the repo**.
- **Impact:** the wave would be blind on PostHog funnels, heatmaps, and **session replay** (so masked-replay
  privacy can't be verified in rendered output while dark). Server-side `recommendation_impressions` + Sentry
  still work, so it is not *fully* blind.
- **Not** a security/privacy/data risk — it fails safe (collecting nothing ⇒ no PII).
- **Action (required-before-invite):** confirm `VITE_POSTHOG_KEY` is set in the prod env; confirm one real
  event lands + one replay shows masked text — **or** consciously accept server-side-only observability for
  the wave and record that decision.

### 8b. `lists` anon posture — **P2 (hygiene + product decision; not a blocker)**
`lists` (and `list_movies`) carry full anon table grants (`SELECT/INSERT/UPDATE/DELETE/TRUNCATE/...`), and a
`"Public lists are readable by anyone"` policy (`roles=public`, `is_public=true OR auth.uid()=user_id`).
Firsthand analysis:
- **anon cannot write** — the only write policy (`"Users can manage own lists"`, ALL) requires
  `auth.uid()=user_id`, which is NULL for anon ⇒ all anon INSERT/UPDATE/DELETE are blocked by RLS. The anon
  write **grants are inert** (hygiene risk only, like the S1.3 `beta_members` latent grants).
- **No private-data leak** — `is_public` **defaults to `false`** (verified), 0 private lists exist, and anon
  can read only explicitly-public lists. **0 lists are tester-owned** (the 4 public lists are dev/curated).
- **The residual:** explicitly-public lists are readable pre-auth (AppShell doesn't gate `/lists`), and the
  inert anon write grants are untidy.
- **Action (recommended before 5–10; required before public):** a tiny grant-only migration to
  `revoke insert, update, delete, truncate, references, trigger on public.lists, public.list_movies from anon, authenticated`
  (keep authenticated SELECT for owner reads), and decide whether public-list viewing should require auth
  (scope the public-read policy to `authenticated`) or remain share-by-design. **Out of this operational
  phase's scope** (RLS/migration change) → propose as `S1.5` or `B2.4`.
